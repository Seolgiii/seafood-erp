한 눈에 보는 차이

window.alert() toast()

시나리오 1 — 폼 검증 실패

"수량을 입력해주세요" 같은 입력 검증 실패 메시지. 두 폰에서 같은 버튼을 눌러보세요.

window.alert()

입고 신청

ⓘ 수량을 비워두고 등록 누르세요

toast()

입고 신청

ⓘ 수량을 비워두고 등록 누르세요

체감 포인트: alert는 OK 누를 때까지 화면이 멈춰서 키보드도 닫히고 다른 입력도 막힙니다. toast는 메시지가 떠도 다른 필드 클릭·입력이 그대로 가능합니다.

시나리오 2 — 연속 알림 (3번 빠르게)

네트워크가 느린 모바일에서 사용자가 버튼을 여러 번 누르면 같은 검증 알림이 반복됩니다.

window.alert()

알림 3번 띄우기

버튼 한 번만 누르세요. 내부에서 alert 3번 호출됩니다.

매번 OK를 눌러야 다음 alert가 뜹니다 → 결국 OK를 3번 눌러야 함.

toast()

알림 3번 띄우기

버튼 한 번만 누르세요. 내부에서 toast 3번 호출됩니다.

위로 쌓였다가 3초 뒤 알아서 사라집니다. 클릭 한 번도 필요 없음.

체감 포인트: 더블 클릭·재시도 같은 정상 사용자 행동에서도 alert는 사용자에게 추가 작업(OK 클릭)을 강요합니다.

시나리오 3 — 메시지 종류 구분

성공·정보·에러 메시지를 시각적으로 구분할 수 있는지.

window.alert()

알림 종류

모두 똑같은 OS 다이얼로그. 사용자는 색이나 모양만으로 메시지 종류를 알 수 없습니다.

toast()

알림 종류

초록=성공, 파랑=정보, 검정=에러. 색만 보고 즉시 인지 가능.

권장 패턴 — 뭘 어디에 쓸지

✓ toast로

저장 성공, 조회 실패, 네트워크 오류, 자동 로그아웃 알림, 백그라운드 작업 결과 등 일시적·정보성 알림

⚠ 인라인 메시지로

폼 검증 실패(필수 필드 누락 등). 해당 필드 바로 아래에 빨간 텍스트 + 아이콘. 사용자가 어떤 필드를 고쳐야 하는지 즉시 인지.

⛔ 디자인된 모달로

삭제 확인, 결재 반려 사유 입력, 권한 거부 등 사용자 결정·입력이 필요한 무거운 메시지. RejectBottomSheet 같은 자체 컴포넌트 사용.

정리: alert을 무조건 toast로 바꾸는 게 아니라 메시지 무게에 맞게 분리합니다. 현재 코드에서 alert이 가장 많이 쓰이는 곳 = 폼 검증 실패 = 사실 인라인 메시지가 가장 적합한 자리. toast는 신청 성공·네트워크 오류 같은 비차단 알림에 어울립니다.
