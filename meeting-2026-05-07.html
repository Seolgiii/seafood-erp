Part A — 시스템 소개

A.1 SEAERP가 무엇인가

SEAERP는 수산물 입출고 · 재고 · 비용을 관리하는 자체 ERP 시스템입니다.

핵심은 LOT 단위로 재고를 추적하고, 출고 시점에 그 LOT에 들어간 모든 비용을 합산해 손익을 자동으로 계산하는 것. 매입처에서 들여온 수산물이 어디 보관됐고, 얼마에 사서 얼마에 팔렸고, 그 사이에 든 보관 비용이 얼마인지 — 이걸 LOT 하나하나 단위로 따라갈 수 있게 만든 시스템입니다.

왜 만들었나 — 수기/엑셀로는 LOT별 비용 추적과 손익 계산이 너무 번거롭고, 누락이 잦았음. 작업자가 핸드폰에서 입출고 신청하고, 관리자가 결재만 누르면 재고가 자동 차감되고 손익까지 계산되는 흐름이 목표.

A.2 데이터의 흐름

Step 1
매입처
수산물을 사오는 거래처
Step 2
입고 신청 → 승인
LOT 자동 생성, 보관처 비용 적용
Step 3
보관처
냉동창고 등에서 보관, 보관일수만큼 냉장료 누적
Step 4
출고 신청 → 승인
재고 차감, 출고시점 비용 7개 필드 스냅샷, 손익 계산
Step 5
거래처 (출고)
판매 완료, 손익 확정

중간에 보관처를 바꾸고 싶으면 재고 이동으로 처리. 회사 운영비는 별도 지출결의로 관리.

A.3 핵심 용어 사전

이 단어들이 회의 중 계속 나옵니다. 미리 한 번 보고 가시면 좋습니다.

LOT
한 번에 들여온 동일한 품목·규격의 재고 묶음. 모든 추적의 기본 단위.
예: 260507-MC1-11-26-0042
LOT번호
YYMMDD-품목코드-규격-미수-일련번호 형식. 입고 승인 시 자동 생성.
보관처
재고를 보관하는 창고/위치. 보관처마다 냉장료·입출고비·노조비가 다름.
매입처
수산물을 사오는 거래처.
미수
평균 마릿수. 1박스에 평균 몇 마리가 들어있는지를 나타내는 숫자. 품목·규격마다 다름.
예: "26"이면 1박스에 평균 26마리
박스
기본 거래 단위. 입고/출고 모두 박스 수로 처리.
잔여수량
현재 LOT에서 더 출고할 수 있는 박스 수. 출고 승인 시 자동 차감.
결재
신청을 승인 또는 반려하는 작업. 승인 시 재고/비용 자동 처리.
수매가
매입처에서 수산물을 사올 때 지불한 박스당 가격. LOT별로 저장.
판매원가
출고 시점의 단가 + 누적냉장료 + 입출고비 + 노조비 합산. 손익 계산의 기준.
손익
판매금액 − 판매원가. 출고 승인 시점에 자동 계산되어 출고 관리에 저장.
PIN
작업자별 4자리 비밀번호. 로그인 시 사용.

A.4 하루의 흐름

아침 09:00 — 관리자에게 일일 정산 메일 도착 (어제 처리분 + 결재 대기 알림)
  1. 작업자 — 핸드폰으로 SEAERP 접속, PIN으로 로그인. 들어온 수산물은 입고 신청, 나갈 수산물은 출고 신청, 보관처 옮길 건 이동 신청.
  2. 관리자 — 결재 수신함에서 신청 건 확인 후 승인 또는 반려. 100만원 미만 지출은 ADMIN이 즉시 승인 가능, 100만원 이상은 MASTER 권한 필요.
  3. 승인 즉시 자동 처리 — LOT 생성 / 재고 차감 / 비용 적용 / 손익 계산 / PDF 생성 모두 자동.
  4. 잘못 결재한 경우 — 완료 탭에서 카드 클릭해 승인 ↔ 반려 토글 가능. 재고도 자동 복구됨.
  5. 저녁/익일 09:00 — 일일 정산 메일에서 어제 흐름 + 손익 한눈에 확인.
Part B — 기능 설명서

현재 시스템에 구현된 기능을 하나씩. 각 기능의 "무엇을 한다 → 어떻게 사용한다 → 자동으로 처리되는 부분" 순서로 정리.

📥 B.1 입고 (들어오는 재고)

매입처에서 수산물을 사와서 회사 보관처에 들이는 작업.

누가
작업자가 신청, 관리자가 승인
입력 정보
매입처, 매입자, 선박명, 품목, 규격, 미수, 박스 수, 보관처, 수매가, 입고일
승인 시 자동
LOT번호 자동 생성 / 보관처 비용 자동 적용 (냉장료·입출고비·노조비) / 입고증 PDF 자동 발행
반려 시 자동
LOT 재고수량을 0으로 (LOT 자체는 보존) / 비용 필드 클리어
특이사항
승인 ↔ 반려 양방향 변경 가능 — 재승인하면 모든 데이터 자동 복원
📤 B.2 출고 (나가는 재고)

보관 중인 재고를 거래처로 판매 출하하는 작업.

누가
작업자가 신청, 관리자가 승인
검색 방식
LOT번호 또는 품목명으로 검색 → 다건 카트에 담아 묶음 신청
입력 정보
출고일, 거래처, LOT별 박스 수, 판매가
승인 시 자동
LOT 재고 차감 / 잔여수량 차감 / 출고시점 비용 7개 스냅샷 저장 (단가/냉장료/보관일수/입출고비/노조비/판매원가/손익) / 출고증 PDF
반려 시 자동
잔여수량 +복구 / LOT 재고 +복구 / 비용 7개 클리어
왜 스냅샷?
출고한 후에 보관처 비용이 바뀌어도 그 출고 건의 손익은 그대로 보존되어야 하기 때문
🔄 B.3 재고 이동 (보관처 변경)

한 보관처에 있던 재고 일부를 다른 보관처로 옮기는 작업.

누가
작업자가 신청, 관리자가 승인
입력 정보
원본 LOT, 이동 후 보관처, 이동 박스 수
승인 시 자동
새 입고관리/LOT 자동 생성 (이동분, 비고="재고 이동") / 원본 LOT 차감 / 새 LOT의 수매가는 비례 계산 (원본 비용 × 이동수량/원본 재고수량)
주의사항
미구현 반려 자동 복구는 LOT 중복 생성 위험으로 미구현. 발생 시 관리자가 수동 보정 필요. 사용 빈도 낮으므로 의도된 결정.
💸 B.4 지출결의

회사 운영비/잡비 등을 결재로 처리.

누가
작업자가 신청, 관리자(ADMIN/MASTER)가 승인
입력 정보
지출일, 계정과목, 금액, 사용 내역, 영수증 (사진 업로드)
권한 분리
100만원 미만 → ADMIN(관리자) 승인 가능
100만원 이상 → MASTER(전결권자) 승인 가능
승인 시 자동
지출결의서 PDF 자동 발행
✅ B.5 결재 시스템

신청 건을 검토해 승인 또는 반려하는 통합 도구.

화면 구성

🗃️ B.8 마스터 데이터 (시스템의 뼈대)

입고/출고/이동/지출이 모두 참조하는 기준 데이터. 마스터가 정확하지 않으면 모든 흐름이 어긋남.

👤 B.9 작업자 / PIN / 권한
로그인

권한 단계

Part C — 운영 결정 의제

시스템 자체보다 "우리가 어떻게 운영할 것인가"에 대한 의사결정 항목들. 각 의제마다 왜 결정해야 하는지 + 무엇을 결정할지를 분리해서 정리.

의제 1

마스터 데이터 — 왜 지금 확정해야 하는가

현재 상황: 갈치 품목코드가 미정이어서 갈치 입고 16건의 LOT번호가 생성되지 않은 채 누적되어 있음. 200건 가량 옛 LOT에 보관처 비용이 누락된 상태일 가능성도 있음.

왜 확정해야 하는가:

  • 품목코드가 없으면 LOT번호 생성이 안 되어 추적 불가
  • 보관처 비용이 누락된 LOT는 출고 시 손익 계산이 부정확 (특정 비용이 0으로 처리됨)
  • 운영 시작 후엔 매일 데이터가 쌓여서 "잘못된 시작값" 보정이 점점 어려워짐 — 지금이 cleanup 적기
🎯 결정해야 할 것
  • 운영에 사용할 품목 리스트 확정 — 현재 품목마스터에 등록된 것 외에 추가/삭제할 품목?
  • 품목코드 체계 — 현재 "MC1" 같은 약어. 갈치는 어떤 코드로? 누가 결정?
  • 신규 품목 등록 절차 — 작업자가 자유 입력 가능 vs 사전 마스터 등록만 허용?
의제 2

보관처 운영 정책

현재 상황: 보관처 비용 이력이 7~9건만 입력되어 있는 상태. 실제 회사가 거래하는 보관처가 정확히 몇 개인지, 비용 이력 누락이 얼마나 되는지 불명확.

왜 정리해야 하는가:

  • 비용 이력이 없는 보관처에 입고하면 LOT의 비용 필드가 비어서 → 출고 시 손익 계산 부정확
  • 사용 안 하는 보관처가 마스터에 남아있으면 작업자가 잘못 선택할 수 있음
  • 보관처별 단가 변경 정책이 불분명하면 "옛날 LOT vs 새 LOT" 구분이 모호해짐
🎯 결정해야 할 것
  • 실제 사용 보관처 수 — 7~9개가 맞나? 더 많은가? Airtable 직접 확인 필요
  • 비활성 보관처 정리 — 사용 안 하는 보관처 마스터에서 비활성/삭제할지
  • 비용 단가 변경 정책 — 단가 바뀔 때 누가 비용 이력 행을 새로 추가? 적용시작/종료일은 누가 관리?
  • 새 보관처 추가 절차 — 신규 계약 시 ① 마스터 등록 → ② 비용 이력 입력 (이 두 단계 누가 책임?)
의제 3

보관처 비용 이력 정리

구조 설명: 보관처 비용 이력은 "냉장료, 입출고비, 노조비 3가지를 기간(시작일~종료일)별로 저장"하는 표. 입고 시 SEAERP가 자동으로 "이 LOT의 입고일에 적용되던 비용은 무엇인가"를 찾아서 LOT에 박아둠.

현재 누락 가능 상태:

  • 지금 LOT 200건 가량이 보관처 비용이 안 박혀 있을 가능성 (4월 메모)
  • 4/27에 보관처 Link 통일 작업 후 일부 자동 해소 가능성 — 미확인
  • 동결비는 코드상 출고 비용 계산에 등장하지만 보관처 비용 이력 표에 컬럼이 없음. 별도로 어디에 입력하는 필드인지 정책 부재.
🎯 결정해야 할 것
  • 현재 비용 이력 7~9건 vs 실제 필요한 수 — 보관처 × 시기 조합으로 몇 건이 정상?
  • 누락 LOT 200건 보정 책임 — 일괄 업데이트 누가? 어떤 기준?
  • 동결비 정책 — 별도 마스터 없이 LOT마다 수기 입력? 아니면 새 마스터 만들어서 자동 적용?
  • 비용 이력 입력 책임자 — 단가 협상 후 시스템 입력까지 누가 담당?
의제 4

매입처 마스터 운영

현재: 매입처가 마스터로 분리되어 있어 입고 시 선택 가능. 매입자/선박명도 함께 추적 가능한 구조.

매입처 정보가 정확하지 않으면 추후 거래처 분석(어디서 가장 많이 사오나, 매입처별 평균 단가 등)이 불가.

🎯 결정해야 할 것
  • 현재 등록된 매입처 수 확인 + 활성/비활성 정리
  • 신규 매입처 발생 시 절차 — 사전 등록 강제 vs 입고 시 즉석 등록 허용
  • 관리할 매입처 정보 범위 — 이름만? 대표자/연락처/사업자번호도?
  • 선박명 마스터화 여부 — 현재 자유 입력. 선박 정보 마스터 도입 시점?
의제 5

품목마스터 — 갈치 사례 처리

현황: 갈치 품목코드 미확정 → LOT 16건이 LOT번호 없이 입고됨.

고려 사항:

  • 이 16건은 출고는 가능하나 LOT번호로 추적이 어려움
  • 품목코드 정한 후 16건에 사후 LOT번호 부여 vs 그대로 두기 양자택일
  • 이 사례는 "신규 품목 발생 시 즉석 처리 vs 사전 절차" 원칙을 정해두면 재발 방지
🎯 결정해야 할 것
  • 갈치 품목코드 확정 (예: GC, GAL 등)
  • 16건 사후 처리 방안 — 수기 LOT번호 부여? 비고 마킹 후 그대로?
  • "신규 품목" 등록 표준 절차 — 누가, 어떤 정보로, 어디에 알려야 하나
의제 6

결재 정책 — 권한 / 임계값 / 양방향 변경

현재 정책: 100만원 미만 지출은 ADMIN, 100만원 이상은 MASTER가 즉시 승인 가능. 결재 후에도 양방향 변경(승인↔반려 토글) 가능.
🎯 결정해야 할 것
  • 100만원 임계값 — 회사 규모에 적정한가? 50만원/200만원 등 다른 기준?
  • MASTER 권한자 — 현재 누가 보유? 부재 시 대체 권한 정책?
  • 반려 사유 작성 가이드라인 — 단순 "내용 부적절" 금지, 구체적 사유 강제 등 규칙?
  • 양방향 변경 권한 범위 — 모든 관리자 가능 vs 마스터만 가능? (실수 시 데이터 의도와 다르게 변경됨)
의제 7

작업자 / PIN 운영 정책

보안 기능: PIN scrypt 해시화 + 5회 실패 시 5분 잠금, 다시 5회 시 30분 잠금. 자동 로그아웃 5분 전 경고 배너.
🎯 결정해야 할 것
  • 신규 작업자 등록 절차 — PIN 발급은 누가? 권한 부여 기준은?
  • 퇴사 작업자 처리 — Airtable에서 비활성 vs 완전 삭제? (과거 신청 기록 보존 위해 비활성 권장)
  • PIN 분실/잠금 대응 — 30분 기다림 vs 관리자가 즉시 잠금 해제? 잠금 해제는 누가?
  • PIN 정기 변경 — 현재 변경 정책 없음. 도입 여부?
  • PIN 잠금 안내 — 작업자에게 "5회 실패 시 잠금"을 어떻게 사전 안내?
의제 8

데이터 정합성 점검 책임

왜 필요한가: 시스템이 자동으로 처리하지만 100% 보장은 아님. 결재 처리 중 일시적 네트워크 오류, 동시 작업으로 인한 충돌 등 드물게 발생. 발생 시 알림이 시스템 로그에 남고, 누군가 보고 수동 보정해야 함.

정기 점검 없이 운영하면 작은 오류가 쌓여서 손익 계산이 점점 부정확해지는 위험.

🎯 결정해야 할 것
  • 일일 점검 담당자 — 매일 누가 시스템 알림 확인? 관리자/마스터?
  • 점검 시각 — 일일 보고서 도착(09:00) 후 09:30? 다른 시각?
  • 이상 발견 시 대응 채널 — 카톡/슬랙/메일 중 어디로 공유?
  • 월간 정기 점검 — Airtable 직접 들여다보면서 LOT 누락/비용 누락 확인하는 루틴 도입?
  • 백업 정책 — Airtable 자체 백업 외에 별도 외부 백업 필요한가?
의제 9

일일 보고서 운영

현재: 매일 09:00 KST에 어제 처리분 + 결재 대기 + 손익 추정이 이메일로 발송. 5/7 도메인 mismatch 이슈 발견 후 본문 링크 제거하여 deliverability 개선 시도 중.
🎯 결정해야 할 것
  • 수신자 추가 여부 — 현재 1명? 마스터까지 포함? 회계 담당까지?
  • 도착 안 됐을 때 알아채는 방법 — 받지 못하면 누가 신고? 일주일 안 와도 모를 위험?
  • 추가 항목 우선순위 — 반려 건수 / 데이터 누락 LOT 건수 / 잠금 발생 사용자 — 어떤 게 가장 중요?
  • 보고서 시간대 — 09:00이 적정한가? 출근 직후 너무 이른가?
의제 10

향후 우선순위 (비즈니스 관점)

현재 SEAERP는 입출고/이동/지출 + 결재 + 보고가 갖춰진 1차 완성 단계. 1~2주 운영 후 다음 단계로 무엇을 우선할지가 회사 운영 효율성에 큰 영향.
🎯 결정해야 할 것 (1주 운영 후 회고 시 함께)
  • FIFO 평단가 시스템 — 같은 품목·규격이라도 입고 시기별 평단가 자동 계산. 도입 우선도?
  • 부자재 재고 확장 — 포장지, 아이스팩 등도 SEAERP에서 관리?
  • 바코드/QR 출고 자동화 — 현재 QR로 LOT 호출까지. 출고 자동 완료까지 확장?
  • 팩스 자동 발송 — 거래처에 자동으로 거래명세서 팩스 전송 기능 도입 우선도?
  • 기기별 고정 로그인 — 매번 PIN 입력 대신 기기 인증으로 자동 로그인?
  • 안정화 vs 신규 기능 — 1주차 발견 이슈 보강 vs 신규 기능 추가 — 어디에 시간 더 쓸지?
