# D4 TRANSFER 반려 자동 복구 (안전 가드)

## 트리거
재고 이동(TRANSFER) 결재가 반려될 때 (`admin.ts:updateApprovalStatus` → `transfer.revertTransferOnReject`)

## 흐름 (자동 복구 + 안전 가드 3종)
1. 관리자가 TRANSFER 결재 반려 → [[30_모듈별_상세/결재_수신함]]
2. `revertTransferOnReject` 호출 → 안전 가드 3종 검사:
   - (a) 신규 LOT.재고수량 == 이동수량 (이동 후 출고/조정 없음)
   - (b) 신규 LOT을 원본으로 한 활성 재이동(반려 외) 없음
   - (c) 신규 입고관리에서 활성 출고(반려 외) 없음
3. 통과 시 자동 복구:
   - 원본 LOT.재고수량 += 이동수량
   - 원본 입고관리.잔여수량 += 이동수량
   - 신규 LOT.재고수량 = 0 (soft delete, 레코드 보존)
   - 신규 입고관리.잔여수량 = 0, 승인상태 = "반려"
   - 재고 이동.승인상태 = "반려"
4. 가드 하나라도 실패 시 [INTEGRITY-ALERT] 로그 + `success:false` → 반려 처리 자체 차단 → 운영자가 수동 보정 후 재시도

- [[30_모듈별_상세/재고_이동]] → [[30_모듈별_상세/입고_관리]]: 원본 잔여수량 +이동수량 / 신규 잔여수량 0·승인상태 반려
- 가드 실패 시 [[30_모듈별_상세/재고_이동]] → [[30_모듈별_상세/운영_로거]]: [INTEGRITY-ALERT] prefix 로그

## 옵션 B 이월 4개의 처리
- 신규 LOT의 이월냉장료/이월입출고비/이월노조비/이월동결비는 재고수량이 0이 되면 손익 계산상 의미 없음 → 별도 클리어 불필요
- 원본 LOT의 이월값은 이동 시 손대지 않으므로 그대로 유지 (코드 의도)

## 안전 가드 차단 시 운영자 행동
1. INTEGRITY-ALERT 로그에서 차단 사유 확인 (출고 발생 / 재이동 / 재고 변동)
2. 차단 원인이 된 후속 작업(출고·재이동)을 먼저 반려 또는 정리
3. 다시 TRANSFER 반려 시도
